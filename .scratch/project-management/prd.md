# PRD: 项目管理功能

## Problem Statement

MES系统需要实现项目管理功能,用于管理产品型号系列(如iPhone 17)。项目是MES系统主数据的顶层分类,料号(具体产品变体)需要归属于项目。目前系统缺少项目管理模块,无法对产品型号系列进行分类管理,导致后续的料号管理、工单管理等业务无法正常开展。

## Solution

实现一个完整的项目管理模块,包括项目的增删改查、状态管理、批量删除和导出功能。项目作为主数据,具有以下特性:
- 独立存在,不依赖其他业务数据
- 可随时启用/禁用,禁用后不影响已有数据但阻止新料号创建
- 删除时需检查是否已有料号关联,有料号的项目禁止删除
- 项目编码创建后不可修改,保证业务标识稳定性

项目数据支持分页查询、模糊搜索、状态筛选和Excel导出,为后续料号管理、工单管理等功能提供基础数据支撑。

## Implementation Decisions

### 数据库设计

创建 `mes_project` 表,字段包括:
- id: BIGINT主键,自增
- project_code: VARCHAR(50),项目编码,唯一索引
- project_name: VARCHAR(100),项目名称
- description: VARCHAR(500),项目描述
- enable: TINYINT,启用状态(默认1)
- remark: VARCHAR(500),备注
- 审计字段: created_by, create_time, updated_by, update_time
- deleted: TINYINT,逻辑删除标记

编码字段设置唯一索引 `uk_project_code`。

### 后端模块

创建以下模块(遵循ADR-0008基础数据模块开发规范):

**实体层**: `MesProject` 实体类
- 使用MyBatis-Plus注解映射数据库表
- 审计字段使用 `@TableField(fill = FieldFill.INSERT/UPDATE)` 自动填充
- 逻辑删除使用 `@TableLogic` 注解

**数据访问层**: `ProjectMapper` 接口
- 继承 `BaseMapper<MesProject>`
- 无需自定义方法,使用MyBatis-Plus内置方法

**业务逻辑层**: `ProjectService` 接口及实现 `ProjectServiceImpl`
- 方法设计参考 `ProcessService`:
  - `page(ProjectQueryRequest)` - 分页查询
  - `add(ProjectAddRequest)` - 新增项目(包含编码唯一性校验)
  - `edit(ProjectEditRequest)` - 编辑项目
  - `delete(Long id)` - 删除项目(包含料号关联检查)
  - `batchDelete(List<Long> ids)` - 批量删除(包含料号关联检查和结果汇总)
  - `updateStatus(Long id, Integer enable)` - 更新启用状态
  - `export(ProjectQueryRequest, HttpServletResponse)` - 导出Excel

**Controller层**: `ProjectController`
- REST API路径: `/api/project`
- 方法设计参考 `ProcessController`:
  - `POST /api/project/page` - 分页查询
  - `POST /api/project` - 新增项目
  - `PUT /api/project/{id}` - 编辑项目
  - `DELETE /api/project/{id}` - 删除项目
  - `DELETE /api/project/batch` - 批量删除
  - `PUT /api/project/{id}/status` - 修改状态
  - `GET /api/project/export` - 导出数据

**DTO对象**:
- `ProjectQueryRequest` - 查询请求(code, name, enable, pageNum, pageSize)
- `ProjectAddRequest` - 新增请求(code, name, description, remark)
- `ProjectEditRequest` - 编辑请求(id, name, description, remark)
- `ProjectResponse` - 响应对象(完整字段)
- `ProjectPageResult` - 分页结果(列表字段)

### 权限配置

预置以下权限到 `sys_permission` 表:
- `project:read` - 查看项目
- `project:add` - 新增项目
- `project:update` - 编辑项目
- `project:delete` - 删除项目
- `project:export` - 导出项目

权限采用扁平化命名 `{模块}:{操作}`,符合CONTEXT.md定义的权限体系。

### 业务逻辑约束

**编码唯一性校验**:
新增项目时,检查数据库是否存在相同编码的项目,若存在则抛出 `BusinessException("项目编码已存在")`。

**删除保护逻辑**:
- 单个删除: 检查项目下是否已有料号(通过料号表的project_code字段查询),若有则抛出 `BusinessException("该项目下已有料号,无法删除")`
- 批量删除: 逐个检查料号关联,跳过有料号的项目,最后汇总结果。若有跳过项,抛出 `BusinessException("成功删除X个项目,跳过Y个项目(已关联料号)")`

**状态切换**:
可随时切换启用/禁用状态,无需检查关联数据。禁用项目后,新料号无法关联该项目,但已有料号和工单不受影响。

**导出实现**:
使用EasyExcel库导出Excel文件,文件名格式: `项目列表_YYYYMMDD_HHmmss.xlsx`

### 前端交互细节

**描述字段显示**:
表格中描述字段过长时(超过30字符),显示前30字符加省略号,鼠标悬停时通过Tooltip显示完整描述。

**批量删除结果**:
批量删除后,通过 `ElMessage` 提示成功和跳过数量,若有跳过项则显示详细原因。

**状态切换**:
使用 `el-switch` 组件,切换时调用API并刷新列表,无权限时显示为禁用状态或Tag标签。

## Out of Scope

以下内容不在本次PRD范围内:

1. **料号管理**: 本次只实现项目管理,料号管理后续单独开发
2. **项目与料号的联动展示**: 项目列表不显示料号数量或料号列表,料号管理页面单独展示
3. **项目统计分析**: 不实现项目维度的统计分析功能(如每个项目的工单数量、SN数量等)
4. **项目类型字段**: 项目不设置类型字段,后续如有需求可扩展
5. **项目状态字段**: 项目不设置研发中/量产中/已停产等业务状态,仅使用启用/禁用控制
6. **项目负责人字段**: 项目不设置负责人字段
7. **项目时间范围**: 项目不设置开始日期、结束日期字段
8. **富文本编辑**: 描述和备注字段使用普通文本,不支持富文本格式
9. **项目图片/附件**: 项目不支持上传图片或附件
10. **项目审批流程**: 项目创建、修改、删除不需要审批流程

### 后续依赖模块

项目完成后,后续需要依赖项目数据的模块:
- **料号管理**: 料号需要关联项目(project_code字段)

- **SN管理**: SN通过工单间接关联项目
- **报表统计**: 可能需要按项目维度统计生产数据

### 数据迁移考虑

如果系统已有历史数据需要迁移:
- 需要准备项目数据迁移脚本
- 项目编码需要符合新系统规范(字母、数字、下划线、中划线)
- 迁移后需要验证编码唯一性

### 性能考虑

项目数据量预估:
- 典型制造企业项目数量: 100-500个
- 数据量不大,无需特殊性能优化
- 分页查询默认pageSize=10足够
- 导出功能支持全量导出(不分页)

### 用户体验细节

**编码输入辅助**:
- 新增表单中编码输入框可添加placeholder提示: "如: IPHONE17"
- 编码验证失败时,错误提示应明确说明允许的字符类型

**描述显示优化**:
- 表格中描述字段使用 `el-tooltip` 实现悬停显示完整内容
- 描述过长时截断显示,避免表格列宽过大

**批量操作提示**:
- 批量删除时,确认对话框应显示"确定要删除选中的X个项目吗?"
- 批量删除后,若有跳过项,应明确提示跳过原因

### 与工序管理的对比

项目管理与工序管理的主要差异:

| 特性 | 工序管理 | 项目管理 |
|------|---------|---------|
| 编码修改 | 不可修改 | 不可修改 |
| 删除保护 | 无(工序独立) | 有(检查料号关联) |
| 类型字段 | 有(processType) | 无 |
| 状态字段 | enable字段 | enable字段 |
| 查询方式 | POST /list | POST /page(建议统一) |
| 菜单图标 | Setting | Folder |

### API路径统一

工序管理API使用 `/api/process/add`, `/api/process/edit` 等路径。
项目管理建议使用更规范的RESTful风格: `/api/project` (POST新增), `/api/project/{id}` (PUT编辑)。
后续可考虑统一API风格,但本次遵循工序管理的API设计模式。
