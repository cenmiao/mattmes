# 工序管理模块 PRD

## Problem Statement

离散制造企业在生产过程中有大量的加工站点（如检测站、组装站、包装站等），目前系统缺乏对这些工序站点的统一管理。生产管理员无法在系统中配置和维护工序信息，导致：

1. 无法标准化管理生产线上所有工序站点
2. 无法对工序进行分类和检索
3. 无法追溯产品经过哪些工序
4. 未来无法实现工序级别的权限控制和数据统计

## Solution

在MES系统中新增工序管理模块，提供工序信息的CRUD功能，作为后续生产追溯和过程管理的基础模块。

工序管理模块将：
- 提供工序信息的录入、修改、删除、查询功能
- 支持工序类型的分类管理（检测、组装、包装、其他）
- 支持工序的启用/禁用状态管理
- 支持工序数据的导出功能
- 为未来的"过站记录"模块提供基础数据支撑

## Implementation Decisions

### 领域模型

**工序(Process)** 定义：生产线上所有加工站点的统称，包括检测站、组装站、包装站等，是产品流转的基础节点。

工序类型枚举：
- `INSPECTION` (检测)
- `ASSEMBLY` (组装)
- `PACKAGING` (包装)
- `OTHER` (其他)

### 数据库设计

表名：`mes_process`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键，自增 |
| code | VARCHAR(50) | 工序编码，唯一，创建后不可修改 |
| name | VARCHAR(100) | 工序名称，可重复 |
| process_type | VARCHAR(20) | 工序类型，枚举值 |
| description | VARCHAR(500) | 工序描述 |
| enable | TINYINT | 启用状态：1启用/0禁用，默认1 |
| remark | VARCHAR(500) | 备注 |
| created_by | VARCHAR(50) | 创建人 |
| create_time | DATETIME | 创建时间 |
| updated_by | VARCHAR(50) | 更新人 |
| update_time | DATETIME | 更新时间 |
| deleted | TINYINT | 逻辑删除：0未删除/1已删除，默认0 |

索引：
- UNIQUE INDEX on `code`
- INDEX on `process_type`
- INDEX on `enable`

### API接口设计

| 功能 | 接口路径 | 方法 | 权限 |
|------|---------|------|------|
| 查询列表 | `/api/process/list` | POST | 无需权限 |
| 新增工序 | `/api/process/add` | POST | `process:add` |
| 编辑工序 | `/api/process/edit` | PUT | `process:edit` |
| 删除工序 | `/api/process/delete/{id}` | DELETE | `process:delete` |
| 批量删除 | `/api/process/batchDelete` | DELETE | `process:delete` |
| 更新状态 | `/api/process/status/{id}` | PUT | `process:edit` |
| 导出数据 | `/api/process/export` | GET | `process:export` |

### 数据校验规则

| 字段 | 校验规则 |
|------|---------|
| 工序编码 | 必填，最大50字符，仅允许字母、数字、下划线、中划线，全局唯一 |
| 工序名称 | 必填，最大100字符 |
| 工序类型 | 必填，枚举值校验 |
| 启用状态 | 必填，值域：0或1，新增时默认1 |
| 描述 | 选填，最大500字符 |
| 备注 | 选填，最大500字符 |

### 前端设计

**路由路径**：`/process`

**菜单结构**：
- 父级菜单：生产管理
- 子菜单：工序管理

**页面布局**：
1. 查询区域：工序编码、工序名称、工序类型、启用状态、查询/重置按钮
2. 操作区域：新增、删除、导出按钮
3. 数据表格：复选框、工序编码、工序名称、工序类型、启用状态、描述、备注、创建人、创建时间、操作（编辑、删除）
4. 新增/编辑弹窗：工序编码、工序名称、工序类型、启用状态、描述、备注

**交互特性**：
- 启用状态使用Switch开关，点击立即切换，无需确认
- 删除操作需要确认提示
- 批量删除需要确认提示
- 工序编码创建后不可修改
- 工序类型新增时无默认值必填，编辑时默认当前值可修改
- 导出文件名格式：`工序数据_YYYYMMDD_HHMMSS.csv`

### 代码模块划分

**后端**（mattmes-web模块）：
- Controller层：ProcessController
- Service层：ProcessService、ProcessServiceImpl
- Mapper层：ProcessMapper
- Entity层：Process
- DTO层：ProcessQueryDTO、ProcessAddDTO、ProcessEditDTO、ProcessVO

**前端**（mattmes-ui）：
- 页面组件：src/views/process/ProcessList.vue
- API接口：src/api/process.ts

### 权限配置

权限编码：
- `process:add` - 新增工序
- `process:edit` - 编辑工序、更新状态
- `process:delete` - 删除工序、批量删除
- `process:export` - 导出工序数据

## Testing Decisions

### 测试策略

采用编译验证为主的测试策略：
- 后端：编写关键的Service层单元测试，确保业务逻辑正确
- 前端：通过实际页面操作验证功能
- 重点验证：编译通过、接口可调用、权限控制正确

### 测试重点

**后端单元测试**（ProcessServiceTest）：
- 测试新增工序的成功场景
- 测试新增工序编码重复的场景
- 测试编辑工序的成功场景
- 测试删除工序的成功场景
- 测试批量删除的成功场景
- 测试更新状态的成功场景
- 测试查询列表的分页和筛选功能

**测试风格参考**：
- 参考 UserServiceTest 的测试结构
- 使用 Mockito 模拟依赖
- 测试外部行为，不测试实现细节
- 每个测试独立运行，使用 @BeforeEach 初始化

### 测试范围

**需要测试**：
- ProcessService 的核心业务方法
- 数据校验逻辑
- 异常处理逻辑

**不需要测试**：
- Getter/Setter方法
- MyBatis-Plus的基础CRUD（框架已保证）
- Controller层（通过前端测试验证）

## Out of Scope

本次PRD不包含以下内容：

1. **工序与产线的关联**：未来可能需要"产线管理"模块，工序可以归属到产线
2. **工序与产品的关联**：未来可能需要"产品管理"模块，不同产品经过不同的工序路线
3. **过站记录**：未来需要"过站管理"模块，记录产品经过工序的历史
4. **工序权限细化**：当前只控制CRUD权限，未来可能需要控制到工序级别的数据权限
5. **工序排序**：当前工序没有顺序概念，未来在产线中可能需要定义顺序
6. **工序标准工时**：未来可能需要记录每个工序的标准工时
7. **工序物料关联**：未来可能需要记录工序消耗的物料
8. **工序设备关联**：未来可能需要记录工序使用的设备

## Further Notes

### 数据迁移

新模块，无需数据迁移。

### 回滚方案

如果需要回滚：
1. 删除数据库表 `mes_process`
2. 删除后端代码相关文件
3. 删除前端代码相关文件
4. 删除权限配置数据
5. 删除菜单配置数据

### 性能考虑

- 工序数据量预计不大（几百到几千条），无需特殊优化
- 查询接口使用分页，避免一次性加载大量数据
- 编码字段有唯一索引，查询性能有保障

### 扩展性考虑

- 工序类型使用枚举，方便扩展新类型
- 表结构预留了描述和备注字段，方便补充信息
- 使用逻辑删除，数据可恢复
- API设计遵循RESTful规范，方便扩展

### 与现有模块的关系

- **用户权限模块**：工序管理使用现有的权限体系
- **审计日志**：工序操作会自动记录审计字段
- **后续模块**：工序是"过站管理"等后续模块的基础数据
