# MES系统领域上下文

## 业务定位

**制造类型**: 离散制造

**典型场景**: 电子产品代工生产(如苹果手机代工)

## 核心概念模型

### 核心概念清单

| 概念 | 定义 | 类型 |
|------|------|------|
| **项目** | 产品型号系列，如 iPhone 17 | 主数据 |
| **料号** | 具体产品变体（成品料号），如 iPhone 17 金色 5G版 | 主数据 |
| **工序** | 生产加工站点（检测/组装/包装等），是生产基础单元 | 主数据 |
| **路由** | 工序序列，定义产品生产工艺路线 | 主数据 |
| **工单** | 生产任务，指定生产某料号的某批产品 | 业务单据 |
| **SN** | 序列号(Serial Number)，产品唯一追溯键 | 业务数据 |
| **过站记录** | SN经过工序的日志 | 业务数据 |

### 核心依赖关系图

```
项目 (1:N) 料号 (1:1) 路由 (1:N,有序) 工序
              ↓
           (1:N)
              ↓
            工单 (1:N) SN (1:N) 过站记录
```

### 实施顺序

**第一阶段：主数据**
1. **工序管理** - 基础单元，独立存在
2. **项目管理** - 顶层分类，独立存在
3. **料号管理** - 依赖项目
4. **路由管理** - 依赖工序和料号

**第二阶段：业务单据**
5. **工单管理** - 依赖料号
6. **SN管理** - 依赖工单（SN生成模块）

**第三阶段：业务执行**
7. **过站管理** - 依赖SN、工序、路由

## 核心业务规则

### 项目与料号

| 术语 | 定义 |
|------|------|
| **项目** | 产品型号系列，如 iPhone 17 是一个项目 |
| **料号** | 项目下的具体产品变体，如 iPhone 17 金色 5G版、iPhone 17 白色 256G 美版等 |
| **关系** | 一个项目包含多个成品料号（1:N），一个料号只属于一个项目 |

### 料号与路由

| 术语 | 定义 |
|------|------|
| **绑定规则** | 先创建料号，路由创建时绑定料号（1:1） |
| **继承传递** | 料号的路由 → 工单 → SN（所有SN必须按路由工序顺序生产） |

### 路由与工序

| 术语 | 定义 |
|------|------|
| **路由结构** | 路由 = 有序工序序列 + 工序间的流转关系 |
| **流转类型** | 线性顺序（A→B→C→D）或分支选择（A→B→C→返回B或直跳D） |
| **过站规则** | SN必须按路由定义的流转关系过站，不可跳过中间工序 |
| **下一步定义** | 工序间的"下一步去向"在路由中预定义 |

### 工单与SN

| 术语 | 定义 |
|------|------|
| **工单内容** | 必须包含：项目、料号、生产数量、状态 |
| **SN生成** | 在SN生成模块中选择工单后生成，生成时绑定工单（1:1，终身不变） |
| **SN初始状态** | 生成后自动获取路由（工单→料号→路由），下一工序=路由第一道工序 |

### 过站管理

| 术语 | 定义 |
|------|------|
| **过站记录** | 每次过站生成一条日志，记录：工序、状态、时间、操作员等 |
| **状态更新** | 过站后SN记录"下一工序"字段，指向下一个要去的工序 |

## 工序类型

| 类型 | 说明 |
|------|------|
| **检测** | 对产品进行质量检验的工序，记录检测结果（合格/不合格）及检测数据 |
| **组装** | 将零部件装配成完整产品的工序，记录组装人员、物料批次等追溯信息 |
| **包装** | 产成品包装出货的工序，记录包装人员、包装时间、出货批次等信息 |
| **其他** | 其他类型的工序 |

## 追溯体系

全流程以 **SN** 为核心追溯键，串联：
- 工单信息（项目、料号、生产批次）
- 路由信息（生产工艺路线）
- 过站记录
- 检测数据
- 组装物料批次
- 包装出货信息

实现产品全生命周期可追溯。

## 用户与权限体系

采用**动态角色权限模型**：

| 术语 | 定义 |
|------|------|
| **用户** | 系统使用者，通过**工号**(用户名)标识，可被分配多个角色 |
| **工号** | 用户的唯一标识符，等同于用户名，用于登录认证 |
| **角色** | 由管理员动态创建的权限集合，可灵活命名和配置 |
| **权限** | 最小权限单元，采用扁平化命名（如 `user:add`） |
| **临时Token** | 用户登录后系统自动生成的JWT令牌，有效期6小时，用于前端请求认证 |
| **永久Token** | 由管理员手动生成的长期有效令牌，用于外部系统API调用认证 |

**权限特性**：
- **扁平化**：无分级结构，直接以 `模块:操作` 格式命名
- **按钮级粒度**：每个功能按钮对应一个独立权限
- **多角色叠加**：用户可同时拥有多个角色，权限取并集
- **无角色继承**：角色之间不存在继承关系

**认证方式**：

| 认证通道 | 使用场景 | 认证凭据 | 密码要求 | Token类型 |
|---------|---------|---------|---------|----------|
| **前端登录** | 人类用户通过Vue3界面操作 | 工号 + 密码 | 至少8位，必须包含字母+数字 | 临时Token（6小时） |
| **API调用** | 外部系统/脚本直接调用API | Token | 不涉及 | 临时或永久Token |

**Token验证规则**：
- 临时Token：必须验证有效期（6小时）
- 永久Token：无需验证有效期，长期有效

## 用户管理

**用户状态**：

| 状态 | 说明 | 可登录 |
|------|------|--------|
| **启用** | 正常用户 | 是 |
| **禁用** | 被管理员禁用 | 否 |

**用户信息字段**：
- **必填**：工号（用户名）、密码、姓名
- **选填**：手机号、邮箱

**密码安全策略**：

| 策略 | 规则 |
|------|------|
| **密码强度** | 至少8位，必须包含字母+数字 |
| **首次登录** | 强制修改密码 |
| **密码过期** | 每30天必须修改密码 |
| **错误锁定** | 连续错误5次，锁定10分钟 |

**个人信息维护**：
- 用户可自行维护姓名、手机号、邮箱等非敏感信息
- 工号不可修改

**自动禁用规则**：
- 用户连续30天未登录，账户自动禁用
- 管理员可手动启用被自动禁用的账户

## 登录并发控制

**单终端限制**：
- 同一用户同一时间只能在一个终端登录
- 新登录时检测到已有在线会话，提示"账号已在其他地方登录"
- 用户确认后强制登出旧设备，允许当前设备登录
- 并发登录场景：使用数据库事务锁确保只有一个请求成功

**实现机制**：
- 后端维护 `用户ID → Token` 的绑定关系（数据库存储）
- 新登录时生成新Token并覆盖旧Token
- 旧设备请求时检测Token失效，前端跳转登录页

**Token立即失效场景**：
1. 用户被管理员禁用 → Token立即失效，返回401
2. 用户修改密码 → 旧Token立即失效，强制重新登录
3. 用户在其他设备强制登录 → 旧设备Token立即失效

## 登录日志与审计

**登录日志记录字段**：
- 用户工号、登录时间、登录IP、登录设备信息
- 登录结果（成功/失败）、失败原因（密码错误/账户锁定等）
- 退出时间（仅记录主动退出的情况）

**登录日志管理**：
- 长期保留日志记录
- 提供前端查询页面，供管理员或有权限人员查询
- 支持日志导出功能（用于安全审计）

**特殊情况**：
- 用户直接关闭浏览器，不记录退出时间
- 不使用心跳机制，避免增加系统复杂度

## 数据库设计规范

**审计字段规范**：

所有业务表必须包含以下四个审计字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| **created_by** | VARCHAR | 创建人（记录创建该数据的用户工号） |
| **create_time** | DATETIME | 创建时间 |
| **updated_by** | VARCHAR | 最后修改人（记录最后修改该数据的用户工号） |
| **update_time** | DATETIME | 最后修改时间 |

**适用范围**：系统中所有新建表均需遵守此规范

## 登录模块数据模型

**核心表结构**：

**用户表 (sys_user)**：
- id（主键）
- user_no（工号，唯一索引）
- password（加密存储）
- name（姓名）
- phone（手机号，可选）
- email（邮箱，可选）
- status（状态：1-启用，0-禁用）
- disable_reason（禁用原因：ADMIN_MANUAL-管理员手动禁用，AUTO_INACTIVE-30天未登录自动禁用）
- password_update_time（密码最后修改时间）
- login_error_count（连续登录错误次数）
- lock_time（锁定截止时间）
- last_login_time（最后登录时间）
- current_token（当前登录Token，用于单终端登录控制）
- token_expire_time（Token过期时间）
- 审计字段：created_by，create_time，updated_by，update_time

**角色表 (sys_role)**：
- id（主键）
- role_name（角色名称）
- role_code（角色编码，唯一）
- status（状态：1-启用，0-禁用）
- 审计字段：created_by，create_time，updated_by，update_time

**权限表 (sys_permission)**：
- id（主键）
- permission_name（权限名称）
- permission_code（权限编码，如 user:add）
- 审计字段：created_by，create_time，updated_by，update_time

**用户角色关联表 (sys_user_role)**：
- id（主键）
- user_id
- role_id
- 审计字段：created_by，create_time，updated_by，update_time

**角色权限关联表 (sys_role_permission)**：
- id（主键）
- role_id
- permission_id
- 审计字段：created_by，create_time，updated_by，update_time

**永久Token表 (sys_permanent_token)**：
- id（主键）
- user_id（所属用户）
- token（Token值，唯一索引）
- description（Token用途说明）
- 审计字段：created_by，create_time，updated_by，update_time

**登录日志表 (sys_login_log)**：
- id（主键）
- user_no（工号）
- login_time（登录时间）
- login_ip（登录IP）
- device_info（设备信息）
- login_result（登录结果：1-成功，0-失败）
- fail_reason（失败原因）
- logout_time（退出时间）
- 审计字段：created_by，create_time，updated_by，update_time

## 接口设计规范

**统一响应格式**：
```json
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }
}
```

**标准错误码**：

| 错误码 | 说明 | 触发场景 |
|--------|------|---------|
| 200 | 操作成功 | 正常业务响应 |
| 400 | 业务异常 | 参数错误、密码强度不足、用户名不存在等 |
| 401 | 认证失败 | Token无效、Token过期、被强制登出 |
| 403 | 权限不足 | 无该接口访问权限 |
| 409 | 业务冲突 | 账号已在其他设备登录（需要确认强制登录） |
| 500 | 系统异常 | 后端代码异常 |

**被强制登出响应示例**：
```json
{
  "code": 401,
  "message": "您的账号已在其他设备登录，请重新登录",
  "data": null
}
```

**账号在其他设备登录响应示例**：
```json
{
  "code": 409,
  "message": "账号已在其他地方登录，是否强制登录？",
  "data": {
    "currentLoginTime": "2026-05-30 10:30:00",
    "currentLoginIp": "192.168.1.100"
  }
}
```

## 前端权限控制

**路由模式**：Hash模式（如 `#/user/list`）

**路由分类**：
- **公共路由**：无需认证（`/login`，`/403`，`/404`）
- **认证路由**：需要登录（`/dashboard`，`/change-password`，`/profile`）
- **权限路由**：需要登录+特定权限（如 `/user/list` 需要 `user:list` 权限）

**权限控制层级**：
1. **页面级**：动态路由注册，无权限的路由不注册
2. **菜单级**：侧边栏菜单根据权限动态显示
3. **按钮级**：使用 `v-permission` 指令控制按钮显示

**登录流程**：
```
用户登录 → 后端返回 { token, userInfo, needChangePassword }
→ 存储到 localStorage
→ 检查密码状态（首次登录/过期则跳转修改密码页）
→ 动态注册权限路由
→ 渲染侧边栏菜单
→ 按钮根据权限显示/隐藏
```

## 系统初始化

**默认超级管理员**：
- 工号：`admin`
- 默认密码：`Admin@123`
- 默认角色：`SUPER_ADMIN`（超级管理员）
- 首次登录强制修改密码

**超级管理员特权**：
- 自动拥有所有权限，无需逐一配置
- 后端权限验证拦截器检测到该角色时跳过验证
- 前端禁止删除或禁用超级管理员账号

## 安全规范

**密码加密**：使用 BCrypt 算法（Spring Security默认）

**JWT Token格式**：
- Payload：`{ userId, userNo, exp, iat }`
- 不包含角色权限（避免权限变更后Token不一致）
- 签名密钥通过配置文件管理

**永久Token格式**：使用 UUID，存储在 `sys_permanent_token` 表

## 项目结构

**项目目录**：
- 后端：`mattmes`
- 前端：`mattmes-ui`

**后端模块划分**：
- `mattmes-common`：通用模块（工具类、配置、异常处理）
- `mattmes-system`：系统模块（用户、角色、权限、登录）
- `mattmes-business`：业务模块（SN、过站、检测、组装、包装）

**模块依赖关系**：
```
mattmes-web → mattmes-business → mattmes-system → mattmes-common
```

**后端包命名**：`com.matt.mes.{module}`

**前端技术栈**：Vue3 + TypeScript + Vite + Pinia + Element Plus
