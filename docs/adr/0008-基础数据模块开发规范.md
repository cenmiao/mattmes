# 0008-基础数据模块开发规范

## 状态

已接受

## 背景

MES系统包含多个基础数据模块(工序、项目、料号、路由等),这些模块具有相似的业务特性和技术实现模式。为了:
1. **规范化开发**: 统一代码结构和实现方式
2. **加速开发**: 后续模块可快速复制模式
3. **降低维护成本**: 统一风格便于团队协作和维护

需要制定一套标准化的开发规范。

## 决策

### 1. 适用范围

本规范适用于MES系统的**主数据模块**,特征如下:
- 数据独立性: 不依赖其他业务数据,可独立存在
- CRUD操作: 支持新增、修改、删除、查询
- 启用控制: 具有启用/禁用状态
- 逻辑删除: 使用deleted字段实现软删除
- 审计追踪: 记录创建人、创建时间、修改人、修改时间

当前适用的模块:
- 工序管理 (mes_process)
- 项目管理 (mes_project)
- 料号管理 (mes_material)
- 路由管理 (待开发)

### 2. 数据库设计规范

#### 2.1 表命名规范

- **表名格式**: `mes_{业务实体}`
- **示例**: `mes_process`(工序)、`mes_project`(项目)
- **字符集**: utf8mb4
- **排序规则**: utf8mb4_general_ci

#### 2.2 标准字段

所有基础数据表必须包含以下字段:

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PRIMARY KEY AUTO_INCREMENT | 主键 |
| `enable` | TINYINT | DEFAULT 1 | 启用状态(1=启用,0=禁用) |
| `deleted` | TINYINT | DEFAULT 0 | 逻辑删除(0=未删除,1=已删除) |
| `created_by` | VARCHAR(50) | | 创建人(工号) |
| `create_time` | DATETIME | | 创建时间 |
| `updated_by` | VARCHAR(50) | | 更新人(工号) |
| `update_time` | DATETIME | | 更新时间 |

#### 2.3 业务字段规范

| 字段类型 | 命名规范 | 类型 | 约束 | 说明 |
|----------|----------|------|------|------|
| **编码字段** | `{entity}_code` | VARCHAR(50) | UNIQUE, NOT NULL | 业务编码,创建后不可修改 |
| **名称字段** | `{entity}_name` | VARCHAR(100) | NOT NULL | 业务名称,可修改 |
| **描述字段** | `description` | VARCHAR(500) | | 详细描述 |
| **备注字段** | `remark` | VARCHAR(500) | | 备注信息 |
| **类型字段** | `{entity}_type` | VARCHAR(50) | | 业务类型(如工序类型) |

#### 2.4 索引设计

- **主键索引**: `id`
- **唯一索引**: 编码字段(如 `project_code`)
- **普通索引**: 常用查询字段(如 `enable`, `deleted`)

#### 2.5 建表语句模板

```sql
CREATE TABLE `mes_{业务实体}` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键',
  `{entity}_code` VARCHAR(50) NOT NULL COMMENT '编码',
  `{entity}_name` VARCHAR(100) NOT NULL COMMENT '名称',
  `description` VARCHAR(500) DEFAULT NULL COMMENT '描述',
  `remark` VARCHAR(500) DEFAULT NULL COMMENT '备注',
  `enable` TINYINT DEFAULT 1 COMMENT '启用状态(1=启用,0=禁用)',
  `deleted` TINYINT DEFAULT 0 COMMENT '逻辑删除(0=未删除,1=已删除)',
  `created_by` VARCHAR(50) DEFAULT NULL COMMENT '创建人',
  `create_time` DATETIME DEFAULT NULL COMMENT '创建时间',
  `updated_by` VARCHAR(50) DEFAULT NULL COMMENT '更新人',
  `update_time` DATETIME DEFAULT NULL COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_{entity}_code` (`{entity}_code`),
  KEY `idx_enable` (`enable`),
  KEY `idx_deleted` (`deleted`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='{业务实体}表';
```

#### 3.7 业务逻辑规范

##### 3.7.1 编码唯一性校验

新增时必须校验编码唯一性:
```java
// 检查编码是否已存在
LambdaQueryWrapper<Mes{Entity}> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(Mes{Entity}::getCode, request.getCode());
if ({entity}Mapper.selectCount(wrapper) > 0) {
    throw new BusinessException("编码已存在");
}
```

##### 3.7.2 删除校验逻辑

**单个删除**:
```java
// 检查是否有关联数据
if (hasRelatedData(id)) {
    throw new BusinessException("该{业务实体}下已有{关联数据},无法删除");
}
```

**批量删除**:
```java
int successCount = 0;
int skipCount = 0;
for (Long id : ids) {
    if (!hasRelatedData(id)) {
        {entity}Mapper.deleteById(id);
        successCount++;
    } else {
        skipCount++;
    }
}
if (skipCount > 0) {
    throw new BusinessException(
        String.format("成功删除%d条,跳过%d条(已关联{关联数据})", successCount, skipCount)
    );
}
```

##### 3.7.3 状态切换逻辑

```java
Mes{Entity} entity = {entity}Mapper.selectById(id);
if (entity == null) {
    throw new BusinessException("{业务实体}不存在");
}
entity.setEnable(enable);
{entity}Mapper.updateById(entity);
```

### 5. 权限配置规范

#### 5.1 权限命名规范

权限采用扁平化命名: `{模块}:{操作}`

**标准权限列表**:
| 权限编码 | 权限名称 | 说明 |
|---------|---------|------|
| `{entity}:read` | 查看{业务实体} | 页面访问权限 |
| `{entity}:add` | 新增{业务实体} | 新增按钮权限 |
| `{entity}:update` | 编辑{业务实体} | 编辑按钮、状态切换权限 |
| `{entity}:delete` | 删除{业务实体} | 删除、批量删除按钮权限 |
| `{entity}:export` | 导出{业务实体} | 导出按钮权限 |

#### 5.2 权限初始化

系统初始化时,需要在权限表中预置这些权限数据:
```sql
INSERT INTO sys_permission (permission_name, permission_code) VALUES

('编辑{业务实体}', '{entity}:update'),
('删除{业务实体}', '{entity}:delete'),
```

### 8. 导出功能规范

#### 8.1 后端实现

使用EasyExcel库:
```java
@Override
public void export({Entity}QueryRequest request, HttpServletResponse response) {
    // 查询数据(不分页)
    List<{Entity}Response> list = {entity}Mapper.selectList(buildQueryWrapper(request));

    // 设置响应头
    response.setContentType("application/vnd.openxmlformats-officedocument.spreadsheetml.sheet");
    response.setHeader("Content-Disposition",
        "attachment;filename=" + URLEncoder.encode("{业务实体}列表.xlsx", "UTF-8"));

    // 导出Excel
    EasyExcel.write(response.getOutputStream(), {Entity}Response.class)
        .sheet("{业务实体}列表")
        .doWrite(list);
}
```

```typescript
const handleExport = async () => {

    window.URL.revokeObjectURL(url)
  } catch (error) {
    ElMessage.error('导出失败')
  }
}
```

## 理由

### 为什么制定此规范

1. **提高开发效率**: 新模块开发时可直接套用模板,减少重复设计
2. **保证代码质量**: 统一的代码结构和命名规范,降低出错概率
3. **便于团队协作**: 团队成员可快速理解和维护他人代码
4. **降低维护成本**: 统一风格便于后续重构和优化

### 为什么采用逻辑删除

1. **数据安全**: 误删除后可恢复
2. **审计追踪**: 保留历史数据,便于审计
3. **关联保护**: 避免删除数据导致关联数据异常

### 为什么编码字段创建后不可修改

1. **业务标识**: 编码是业务唯一标识,修改会导致混乱
2. **系统稳定性**: 避免编码变更影响关联数据
3. **用户习惯**: 符合企业管理规范,编码一经确定不轻易变更

## 影响

### 开发流程影响

- 新增基础数据模块时,需按照本规范进行设计和开发
- 代码审查时,需检查是否符合规范要求
- 文档编写时,需参考规范模板

### 团队协作影响

- 团队成员需熟悉本规范
- 代码风格统一,降低沟通成本
- 新成员可通过规范快速上手

### 系统维护影响

- 统一的数据结构便于维护
- 统一的API接口便于前端对接
- 统一的权限模型便于管理
