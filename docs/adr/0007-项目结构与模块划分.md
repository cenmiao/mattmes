# 0007-项目结构与模块划分

## 状态

已接受

## 背景

MES系统需要明确项目结构和模块划分,为后续业务模块扩展预留空间。

## 决策

### 1. 项目目录

**后端项目**: `mattmes`

**前端项目**: `mattmes-ui`

两个项目独立目录,各自独立部署和版本管理。

### 2. 后端模块划分

采用Maven多模块结构:

```
mattmes/
├── mattmes-common/             - 通用模块(工具类、配置、异常处理)
├── mattmes-system/             - 系统模块(用户、角色、权限、登录)
├── mattmes-business/           - 业务模块(SN、过站、检测、组装、包装)
├── mattmes-web/                - Web模块(Controller、启动类)
└── pom.xml                     - 父POM
```

**模块职责**:

| 模块 | 职责 | 依赖 |
|------|------|------|
| **mattmes-common** | 通用组件,可被所有模块依赖 | 无 |
| **mattmes-system** | 系统管理功能(用户、角色、权限) | common |
| **mattmes-business** | MES业务功能(SN、过站等) | common, system |
| **mattmes-web** | Web层,整合所有模块对外提供接口 | system, business |

### 3. 后端包结构

各模块内部包结构:

```
com.matt.mes.{module}           - 模块根包
├── config                      - 配置类
├── controller                  - 控制器(仅web模块)
├── service                     - 业务服务
│   └ impl                     - 服务实现
├── mapper                      - MyBatis Mapper
├── entity                      - 实体类
├── dto                         - 数据传输对象
└── vo                          - 视图对象
```

## 理由

### 为什么采用多模块结构
1. **职责清晰**: 各模块职责明确,便于维护
2. **依赖管理**: business模块依赖system模块,避免循环依赖
3. **扩展性强**: 新增业务模块只需创建新模块,不影响现有代码
4. **按需部署**: 未来可考虑各模块独立部署(微服务演进)

### 为什么单独创建mattmes-web模块
1. **统一入口**: 所有Controller集中在web模块,便于管理
2. **启动类隔离**: 启动类和配置类集中在web模块
3. **依赖整合**: web模块整合system和business,对外提供统一接口

### 为什么前端使用Vite而非Vue CLI
1. **更快的启动**: Vite冷启动极快,无需打包
2. **更快的HMR**: 热更新响应速度快
3. **Vue3官方推荐**: Vue3生态更倾向于Vite

## 影响

### 开发流程影响
- 后端: 在父POM目录下,各模块独立开发和测试
- 前端: 独立项目,通过API与后端交互
- 数据库: 各模块共享同一数据库,表通过模块前缀区分(如 sys_, biz_)

### 版本管理影响
- 后端和前端可独立版本管理
- 建议使用同一Git仓库,通过目录区分
- 或使用两个独立Git仓库
