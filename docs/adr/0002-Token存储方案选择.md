# 0002-Token存储方案选择

## 状态

已接受(临时方案,待切换)

## 背景

系统需要实现以下Token管理功能:
- 单终端登录限制(同一用户只能在一个设备登录)
- 强制登出旧设备
- Token有效性验证(临时Token 6小时有效期)

这些功能需要存储Token与用户的绑定关系。由于当前开发环境无法使用Redis,需要选择替代方案。

## 决策

**当前阶段(开发环境)**: 使用MySQL数据库存储Token
**未来阶段(正式部署)**: 切换到Redis存储

### 数据库方案实现

在 `sys_user` 表添加以下字段:
- `current_token`: 当前登录Token
- `token_expire_time`: Token过期时间

### 代码设计

定义 `TokenStorage` 接口,支持多种实现:
- `DatabaseTokenStorage`: 数据库实现(当前使用)
- `RedisTokenStorage`: Redis实现(未来切换)

通过配置文件控制使用哪种实现:
```yaml
app:
  token:
    storage: database  # 当前,未来切换为 redis
```

## 理由

**选择数据库方案的原因**:
1. 开发阶段并发量低,数据库性能足够
2. 无需额外基础设施,降低开发环境复杂度
3. 实现简单,可快速验证业务逻辑

**未来切换到Redis的原因**:
1. Redis内存存储,读写性能远高于数据库
2. 支持TTL自动过期,无需定时清理
3. 适合高并发场景,减轻数据库压力
4. 支持集群部署,多实例共享Token状态

## 影响

### 当前影响
- `sys_user` 表新增两个字段
- 每次请求验证Token需要查询数据库
- 应用重启不影响Token有效性(数据持久化)

### 切换影响
- 需要部署Redis服务
- 修改配置 `app.token.storage=redis`
- 需要处理已登录用户的Token迁移(或等待自然过期)

## 替代方案

| 方案 | 未选择原因 |
|------|-----------|
| 纯JWT无状态 | 无法实现单终端登录和强制登出 |
| 内存缓存(Map) | 应用重启丢失Token,不适合生产环境 |
| 本地文件存储 | 性能差,不适合并发场景 |
